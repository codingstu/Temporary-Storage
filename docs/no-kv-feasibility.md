# 无 KV 部署可行性梳理

## 目标

在不配置任何 KV（或外部数据库/存储）的前提下，实现“临时分享”的核心体验，并明确可行范围与限制。

## 方案总览

| 方案 | 存储位置 | 跨设备/跨会话可访问 | 服务端是否需要持久化 | 适用场景 | 主要限制 |
| --- | --- | --- | --- | --- | --- |
| 浏览器本地存储 | LocalStorage/IndexedDB | 否（同设备/同浏览器） | 否 | 个人自用、单设备临时保存 | 无法跨设备；清理缓存即丢失；容量受限 |
| 分享链接直传 | URL 携带内容（Base64/压缩） | 是 | 否 | 超短内容/小图 | URL 长度限制；安全性差（明文暴露） |
| 客户端加密后直传 | URL 携带密文 | 是 | 否 | 小体量安全分享 | 仍受 URL 长度限制；大内容不适合 |
| Service Worker + Cache | Cache Storage | 否（同设备） | 否 | 离线缓存、短期复用 | 清理缓存即丢失；无法跨设备 |
| 内存存储（Edge/Node） | 进程内存 | 部分（仅同实例） | 否 | DEMO、临时预览 | 不稳定；实例重启丢失；多实例不一致 |

> 结论：若要支持“跨设备分享链接访问”，没有任何服务端持久化几乎不可行；只能依赖链接自带数据或极短内容。

## 关键差异点

### 1) 跨设备访问
- **必须**有可被访问的共享介质：URL 内容、第三方存储或服务端存储。
- 本地存储方案无法满足跨设备分享的核心体验。

### 2) 内容大小
- URL 传输受限（浏览器/代理对 URL 长度有限制，常见上限为 2KB~8KB 量级）。
- 图片 Base64 体积大，基本不适合 URL 携带。

### 3) 安全性
- 明文 URL 共享会暴露内容。
- 可用“客户端加密+URL 携带密文”提升安全性，但不适合大体量内容。

## 可行的“无 KV 版本”形态

### A. 纯本地模式（推荐用于自用/临时草稿）
- 编辑内容 -> 存 LocalStorage/IndexedDB。
- 不提供跨设备分享链接，仅提供“复制内容”或“下载文件”。
- 支持清空/过期策略（客户端计时）。

### B. 轻量分享模式（极小内容）
- 将内容压缩（如 LZ-String）并 Base64 放入 URL 的 query/hash。
- 可选客户端加密后放入 URL。
- 适合短文本、代码片段，不适合图片。

### C. 内存/实例级分享（临时 DEMO）
- 服务端内存保存，设置 TTL。
- 仅同一实例有效，Vercel 多实例/冷启动会导致不可用。
- 只适合作为“演示/测试”。

## 与当前项目功能的兼容性评估

| 功能 | 无 KV 方案可行性 | 备注 |
| --- | --- | --- |
| 创建分享并生成可访问链接 | 低 | 需要共享介质或外部存储 |
| 图片分享 | 很低 | URL 不适合；本地存储不跨设备 |
| 密码/阅后即焚 | 中 | 可在客户端模拟，但无法跨设备 |
| 访问次数限制 | 低 | 无后端计数难以保证 |
| 时间线/最新内容流 | 很低 | 需要共享存储 |

## 建议结论

- **如果不使用 KV/外部存储**，建议定位为“单机临时编辑器”，去掉分享链接或改为“仅本机可访问”的草稿功能。
- **如果仍要保持跨设备分享**，至少需要一种可配置的后端存储（KV/数据库/对象存储），否则功能会大幅缩水。

## 可选折中方案（最低成本）

- 使用免费的 Upstash Redis（即 Vercel KV 背后服务），只存短期数据并设置 TTL。
- 或使用轻量第三方 Paste 服务（非自建，存在依赖风险）。
